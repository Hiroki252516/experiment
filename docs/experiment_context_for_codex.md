# Experiment Context for Codex: ScoreG-style Local-LLM Coordination Study

この文書は、Codex に対して「このリポジトリで何を作るべきか」だけでなく、**なぜその実験を行うのか、どの設計判断を採用するのか、何をもって成功とみなすのか**を共有するためのコンテキスト仕様書です。

この文書は、`AGENTS.md` や `docs/codex_task.md` の補助ではなく、**Codex が実験意図を誤解しないための背景仕様**として使います。

---

## 1. この実験の目的

このプロジェクトの目的は、**2体のローカル LLM エージェントが、部分的にしか観測できない共有世界の中で、限定的な視覚記号通信だけを用いて協調できるか**を検証することです。

ここで検証したい中心仮説は、次の通りです。

1. 各エージェントが世界の一部しか知らない状況では、協調には通信が必要になる。
2. 通信路を自然言語ではなく **7x7 の二値グリフ（49 bit）** に制限しても、反復試行の中である程度の役割分担や記号的慣習が生じうる。
3. 少なくとも `comm` 条件は `silent` や `random` より高い成績を示すはずである。

本プロジェクトは、**本格的な強化学習研究の完成版を作ること**ではなく、まずは **固定したローカル LLM を使って、協調通信実験の最小基盤を作ること**を目的とする。

---

## 2. 参考にしている研究的発想

この実験は、Foraging Games 系の発想、特に「各エージェントが異なる private information を持ち、より価値の高い資源を協調して選ぶ」タイプの課題に影響を受けている。

ただし、このリポジトリで実装するものは論文の完全再現ではない。以下のように**縮小・単純化したローカル実験版**である。

- 世界を 5x5 の gridworld に限定
- エージェント数は 2
- アイテムは 2 つのみ
- 通信路は 7x7 binary glyph のみ
- 学習はまず重み更新なし
- ローカル実行前提（macOS Apple Silicon + Ollama）

つまり、これは **ScoreG 風の最小再構成実験**である。

---

## 3. 実験のコアアイデア

2体のエージェント `agent_a`, `agent_b` は、**同じ 5x5 の世界**を共有する。

世界には 2 つのアイテムが存在する。
- `left_item`
- `right_item`

各アイテムには 1〜9 の整数価値がある。

ただし、価値情報は各エージェントに分割されており、
- `agent_a` は **left_item の価値のみ**を知る
- `agent_b` は **right_item の価値のみ**を知る

両エージェントは、アイテム位置や各自の位置・相手位置などの**公開情報**は見えるが、相手の private value は直接見えない。

したがって、より高価値なアイテムを共同回収するには、何らかの形で相互に意思疎通する必要がある。

このとき、使える通信は自然言語ではなく、**7x7 の二値グリフ 1 枚だけ**である。

---

## 4. なぜ 7x7 binary glyph を使うのか

通信を自然言語にすると、モデルが既存言語能力を使ってしまい、
「制約下でどのように記号が生まれるか」を見にくくなる。

そのため、本実験では通信路を極端に制限し、各ターンで各エージェントは **49 bit の視覚記号**しか送れないようにする。

この設計によって、少なくとも次のような現象が観察しやすくなる。

- 高価値/低価値の圧縮表現
- 左/右の対象指定
- 待機/進行などの行動指示
- 反復セッションにおける記号の再利用

このグリフは、人間にとって意味不明でもよい。重要なのは、**相手との協調成功率を上げる機能を持つかどうか**である。

---

## 5. 今回の実装対象

今回 Codex に作ってほしいのは、次の最小実験基盤である。

1. PettingZoo `ParallelEnv` ベースの 2 エージェント環境
2. Ollama ローカル API を使う 2 体の LLM エージェント
3. 7x7 glyph を JSON schema 付きで出力するエージェント層
4. `comm / silent / random` の 3 条件比較
5. CSV / JSONL ログ保存
6. 基本解析スクリプト
7. 再現可能な README / docs
8. pytest と smoke test

ここで重要なのは、**まず完全に動く最小版を作ること**である。

## 7. 実験環境の固定前提

- ハードウェア: MacBook Air M4 相当の Apple Silicon macOS

## 8. タスク定義

### 8.1 世界
- 5x5 の gridworld
- エージェント: `agent_a`, `agent_b`
- アイテム: `left_item`, `right_item`
- 最大ステップ数: デフォルト 10

### 8.2 private information
- `agent_a` は left item の価値のみ観測できる
- `agent_b` は right item の価値のみ観測できる

### 8.3 公開情報
- 自分の位置
- 相手の位置
- 2 アイテムの位置
- 現在ステップ数
- 相手の前ターンのグリフ

### 8.4 各ターンで行うこと
各エージェントは毎ターン同時に以下を出力する。

1. 移動アクション 1 つ
2. 7x7 binary glyph 1 枚

### 8.5 行動候補
- `UP`
- `DOWN`
- `LEFT`
- `RIGHT`
- `STAY`
- `PICK`

### 8.6 成功条件
- 両エージェントが同じアイテムを共同回収すること
- そのアイテムが高価値側なら最大報酬

---

## 9. 報酬設計

初期実装では次の単純な設計を採用する。

- 高価値アイテムを共同回収: `+1.0`
- 同じアイテムを共同回収したが低価値側: `+0.2`
- 別々のアイテムを回収: `-0.2`
- 毎ステップ時間罰: `-0.01`

これは、Codex が勝手に複雑化してはならない固定仕様である。

---

## 10. 条件比較

最低限、次の 3 条件を比較する。

### `comm`
エージェントが自分でグリフを生成する通常条件。

### `silent`
グリフを常にゼロベクトルに固定する条件。
通信がない場合のベースライン。

### `random`
グリフをランダム 49 bit に置き換える条件。
「通信チャネルが存在する」だけでは不十分で、「意味ある通信」が必要であることを確認するための比較条件。

Codex はこの 3 条件を runner で切り替え可能に実装すること。

---

## 11. 期待している観察結果

この実験で期待するのは、少なくとも以下の傾向である。

1. `comm` 条件の平均報酬が `silent` と `random` を上回る
2. `comm` 条件で `target agreement rate` が高くなる
3. 反復実行により、特定状況で似たグリフが再利用される兆しが見える

ここでいう「成功」は、記号体系が哲学的に解釈可能であることではなく、**協調行動の成績改善に寄与すること**である。

---

## 12. エージェントの設計思想

各エージェントは、同じベースモデルを使ってよい。

ただし、**内部履歴は必ず分離**しなければならない。
- `agent_a` の memory は `agent_a` 専用
- `agent_b` の memory は `agent_b` 専用

相手の private value を prompt に埋め込んではならない。

このプロジェクトで最重要なのは、**hidden information leakage を絶対に起こさないこと**である。

---

## 13. prompt safety の原則

Codex は以下を厳守すること。

1. `agent_a` 用 prompt に right value を含めない
2. `agent_b` 用 prompt に left value を含めない
3. 相手の推定値を事実のように埋め込まない
4. デバッグログに hidden value を不用意に露出しない
5. テストで leakage を検出する

この安全性は単なる実装上の注意ではなく、**実験妥当性そのもの**である。

---

## 14. セッションメモの扱い

今回はまず「固定 LLM」で始めるため、重み更新による学習はしない。

その代わり、必要なら各エージェントに**private memory**を持たせ、過去エピソードの簡易要約を手元に保持してよい。

ただしこの memory は、あくまでそのエージェント個体に閉じたものとする。
相手エージェントと memory を共有してはならない。

この memory は、厳密な意味での学習ではなく、**セッション内の慣習形成を観察するための補助機構**である。
